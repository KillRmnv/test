**JUnit** и **TestNG** — это специализированные библиотеки (фреймворки) для языка Java, которые созданы для **управления процессом тестирования, его запуска и проверки результатов**.

Их называют _тестовыми движками_. Сама по себе Java не умеет «тестировать» — это язык общего назначения. Без TestNG или JUnit тебе пришлось бы писать логику запуска тестов, проверку условий и сбор отчетов вручную через обычный метод `public static void main`.

## 1. Зачем они нужны? (Решаемые задачи)

Оба фреймворка берут на себя всю рутину по организации тестового процесса:

- **Управление жизненным циклом теста (Аннотации):** Позволяют гибко настраивать, что должно произойти перед запуском теста, во время него и после. Например: открыть браузер $\rightarrow$ запустить 10 тестов $\rightarrow$ закрыть браузер.
    
- **Проверка результатов (Assertions / Утверждения):** Предоставляют готовые методы проверки (например, `assertEquals(expected, actual)`). Если ожидаемый результат не совпал с фактическим, фреймворк аккуратно прерывает тест, помечает его красным и выводит понятную ошибку в консоль.
    
- **Группировка и фильтрация:** Позволяют разделять тесты на логические группы (например, `Smoke`, `Regression`, `Database`) и запускать только нужные в данный момент.
    
- **Параметризация:** Дают возможность запустить один и тот же тест с разными наборами входных данных (например, проверить форму авторизации с 20-ю разными парами логин/пароль), не дублируя код.
    
- **Параллельный запуск:** Позволяют запускать тесты в несколько потоков «из коробки», что критически важно для экономии времени при прогоне сотен UI-тестов.
    

## 2. JUnit vs. TestNG: В чем разница?

Исторически **JUnit** (особенно современный JUnit 5) считается стандартом де-факто для **юнит-тестирования** (модульного тестирования кода разработчиками).

**TestNG** (Test Next Generation) изначально создавался как более мощная альтернатива старым версиям JUnit (JUnit 3/4) и нацелен на **высокоуровневое тестирование** (интеграционное, сквозное E2E и системное), которое чаще пишут QA Automation инженеры.

### Главные отличия:

|**Фича**|**JUnit 5**|**TestNG**|
|---|---|---|
|**Управление через XML**|Нет (запуск настраивается через код/плагины сборщиков).|**Да (testng.xml)**. Можно гибко конфигурировать порядок запуска, параметры и группы прямо в текстовом файле.|
|**Зависимости между тестами**|Не поддерживает (тесты должны быть строго изолированы).|**Поддерживает (`dependsOnMethods`)**. Тест `B` не запустится, если упал тест `A`. Удобно для длинных сквозных сценариев.|
|**Параллельный запуск**|Настраивается через конфигурационные файлы (сложнее).|Настраивается одной строчкой в `testng.xml` (`thread-count="5" parallel="methods"`).|
|**Параметризация**|Реализована через `@ParameterizedTest`.|Реализована через мощный механизм `@DataProvider`.|

## 3. Как это выглядит в коде (Java)

Вот простой пример теста на JUnit 5, который проверяет математическую функцию:

Java

```
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.assertEquals;

public class CalculatorTest {

    private Calculator calculator;

    // Выполняется один раз перед всеми тестами в классе
    @BeforeAll
    public static void globalSetUp() {
        System.out.println("Начало тестирования калькулятора");
    }

    // Выполняется перед КАЖДЫМ тестом (например, для очистки состояния)
    @BeforeEach
    public void setUp() {
        calculator = new Calculator();
    }

    @Test
    @DisplayName("Проверка сложения двух положительных чисел")
    public void testAddition() {
        int result = calculator.add(2, 3);
        // Проверяем, что 2 + 3 действительно равно 5
        assertEquals(5, result, "Сумма рассчитана неверно!");
    }

    @AfterEach
    public void tearDown() {
        calculator = null;
    }
}
```

> 
> _Для Unit-тестирования стандартом является JUnit 5. Для автоматизации UI или API тестирования мы часто выбираем TestNG из-за его удобной конфигурации через XML-файлы, встроенной поддержки параллельного запуска из коробки и возможности задавать зависимости между тестовыми методами»_.
