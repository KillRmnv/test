**Тестирование безопасности (Security Testing)** — это вид нефункционального тестирования, цель которого — найти слабые места, уязвимости и потенциальные угрозы в программном обеспечении, чтобы защитить данные пользователей и саму систему от несанкционированного доступа (хакерских атак, утечек и кражи ресурсов).

Простыми словами: если обычное тестирование проверяет, _что система делает то, что должна_, то тестирование безопасности проверяет, _что система НЕ делает того, чего делать не должна_ (например, не показывает чужие пароли).

## 1. 6 ключевых принципов безопасности (CIA Triad +)

Любая проверка безопасности строится на проверке базовых понятий (особенно важна классическая тройка **CIA**):

1. **Confidentiality (Конфиденциальность):** Доступ к конфиденциальным данным имеют только авторизованные пользователи. (Пользователь А не должен видеть профиль пользователя Б).
    
2. **Integrity (Целостность):** Информация не может быть незаметно изменена или удалена во время передачи или хранения. (Никто не должен перехватить сетевой запрос и подменить сумму перевода).
    
3. **Availability (Доступность):** Авторизованные пользователи имеют постоянный доступ к системе и ее сервисам. (Система должна устоять перед DDoS-атакой).
    
4. **Authentication (Аутентификация):** Система умеет точно подтверждать личность пользователя (проверка пароля, двухфакторка 2FA).
    
5. **Authorization (Авторизация):** Система проверяет права доступа пользователя. (Обычный клиент не должен иметь доступа к админ-панели).
    
6. **Non-repudiation (Апеллируемость / Неотрекаемость):** Пользователь не может заявить, что он не совершал определенное действие в системе, так как все операции логируются.
    

## 2. Главный ориентир: OWASP Top 10

В веб-разработке стандартом де-факто является список **OWASP Top 10** — документ, который регулярно обновляется экспертами по безопасности и содержит 10 самых критичных и распространенных уязвимостей в мире.

Самые известные уязвимости из этого списка, о которых нужно знать:

- **Injections (Внедрение кода / Инъекции):** Хакер вводит вредоносный код в поля ввода, а сервер выполняет его как часть своей программы. Самый яркий пример — **SQL-инъекция**, когда в поле логина вводится строка вроде `' OR '1'='1`, что позволяет войти без пароля или удалить всю базу данных.
    
- **Broken Access Control (Нарушение контроля доступа):** Пользователи получают доступ к ресурсам, которые им не принадлежат. Например, меняя ID в URL-адресе `[site.com/user/101](https://site.com/user/101)` на `[site.com/user/102](https://site.com/user/102)`, можно увидеть личные данные другого человека.
    
- **Cryptographic Failures (Сбои шифрования):** Передача конфиденциальных данных (паролей, номеров карт) в открытом виде (по HTTP вместо HTTPS) или хранение паролей в базе данных без хэширования (или с использованием устаревших алгоритмов вроде MD5).
    

## 3. Основные методы тестирования безопасности

В зависимости от того, как именно мы ищем уязвимости, подходы делятся на несколько видов:

- **Сканирование уязвимостей (Vulnerability Scanning):** Автоматизированный поиск известных уязвимостей с помощью специальных программ-сканеров (например, OWASP ZAP, Nessus).
    
- **Тестирование на проникновение (Penetration Testing / Pen Test):** Имитация реальной хакерской атаки. Специалист по безопасности (этичный хакер) пытается обойти защиту приложения любыми доступными способами, чтобы найти дыры.
    
- **Анализ защищенности кода:**
    
    - **SAST (Static Application Security Testing):** Анализ исходного кода программы на уязвимости без её запуска (White Box).
        
    - **DAST (Dynamic Application Security Testing):** Анализ безопасности запущенного приложения путем симуляции внешних атак (Black Box).

> _«В автоматизации мы можем встраивать базовые проверки безопасности прямо в CI/CD пайплайн. Например, использовать статические анализаторы кода (вроде **SonarQube**) для поиска очевидных дыр в коде (SAST), а также утилиты вроде **OWASP Dependency-Check**, которые при сборке проекта на Java автоматически сканируют наши Maven/Gradle зависимости (`pom.xml` / `build.gradle`) на наличие известных уязвимостей в сторонних библиотеках. Это позволяет поймать проблемы с безопасностью на самых ранних этапах разработки»_.
