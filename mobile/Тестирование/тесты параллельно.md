Параллельный запуск тестов — это главный способ сэкономить время при прогоне автоматических тестов. Вместо того чтобы запускать тесты последовательно один за другим, мы задействуем многопоточность процессора (Thread Pool), сокращая общее время выполнения пакета тестов в несколько раз.

Оба фреймворка поддерживают параллелизацию, но делают это на разных уровнях абстракции.

## 1. Параллельный запуск в TestNG

TestNG изначально проектировался с прицелом на параллельный запуск высокоуровневых UI/API тестов. Настройка выполняется чрезвычайно просто и декларативно — через XML-файл конфигурации.

### Настройка через `testng.xml`

Чтобы запустить тесты параллельно, в тег `<suite>` добавляются два ключевых атрибута: `parallel` и `thread-count`.

XML

```
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "https://testng.org/testng-1.0.dtd">
<!-- Запускаем методы параллельно в 4 потока -->
<suite name="Parallel Test Suite" parallel="methods" thread-count="4">
    <test name="UI Tests">
        <classes>
            <class name="by.bsuir.tests.LoginTest"/>
            <class name="by.bsuir.tests.DashboardTest"/>
        </classes>
    </test>
</suite>
```

#### Возможные режимы параллелизации (`parallel`):

- `parallel="methods"` — каждый тестовый метод (помеченный `@Test`) запускается в отдельном потоке. Самый частый режим.
    
- `parallel="classes"` — все методы внутри одного тестового класса выполняются последовательно в одном потоке, но сами классы запускаются параллельно. Идеально, если тесты внутри класса зависят от общего состояния.
    
- `parallel="tests"` — параллельно запускаются тесты, описанные внутри разных тегов `<test>` в xml-файле.
    

## 2. Параллельный запуск в JUnit 5

В JUnit 5 (начиная с версии 5.3) появилась нативная поддержка параллельного выполнения. По умолчанию она отключена.

### Шаг 1. Настройка конфигурации (`junit-platform.properties`)

Параллелизация в JUnit 5 настраивается глобально для проекта. Для этого в папке ресурсов (`src/test/resources`) нужно создать файл `junit-platform.properties` и добавить туда следующие строчки:

Properties

```
# Включаем параллельное выполнение
junit.jupiter.execution.parallel.enabled = true

# Разрешаем параллелизацию как внутри классов, так и между классами
junit.jupiter.execution.parallel.mode.default = concurrent
junit.jupiter.execution.parallel.mode.classes.default = concurrent

# Настройка стратегии выбора количества потоков (например, динамически по числу ядер процессора)
junit.jupiter.execution.parallel.config.strategy = dynamic
junit.jupiter.execution.parallel.config.dynamic.factor = 1
```

### Шаг 2. Использование аннотации `@Execution` (опционально)

Если параллельное выполнение включено глобально, вы можете точечно управлять конкретными тестовыми классами с помощью аннотации `@Execution`:

Java

```
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.parallel.Execution;
import org.junit.jupiter.api.parallel.ExecutionMode;

// Указываем, что этот класс должен выполняться параллельно
@Execution(ExecutionMode.CONCURRENT)
public class FirstParallelTest {

    @Test
    void testMethodOne() throws InterruptedException {
        System.out.println("Метод 1 запущен в потоке: " + Thread.currentThread().getName());
    }

    @Test
    void testMethodTwo() throws InterruptedException {
        System.out.println("Метод 2 запущен в потоке: " + Thread.currentThread().getName());
    }
}
```

## 🔥 Главная проблема параллельного тестирования: Thread Safety (Потокобезопасность)

Написать тесты и включить параллельный запуск — легко. Намного сложнее сделать так, чтобы они при этом не ломались из-за общей памяти.

### Классический плохой пример в Selenium:

Если у вас есть статическая переменная драйвера `public static WebDriver driver;`, то при запуске в 4 потока все четыре теста попытаются управлять **одним и тем же** окном браузера. Тесты будут перехватывать фокус друг у друга, кликать не туда и падать.

### Как это решать?

1. **Использование `ThreadLocal`:**
    
    Каждый поток должен работать со своим изолированным экземпляром данных (например, своим экземпляром WebDriver).
    
    Java
    
    ```
    public class DriverManager {
        // Оборачиваем драйвер в ThreadLocal
        private static final ThreadLocal<WebDriver> driverThreadLocal = new ThreadLocal<>();
    
        public static WebDriver getDriver() {
            return driverThreadLocal.get();
        }
    
        public static void setDriver(WebDriver driver) {
            driverThreadLocal.set(driver);
        }
    
        public static void unload() {
            driverThreadLocal.remove();
        }
    }
    ```
    
2. **Изоляция тестовых данных:** Тесты не должны использовать одного и того же пользователя на сайте одновременно. Если первый тест меняет пароль пользователю `admin`, а второй в это же время пытается под ним авторизоваться, один из них гарантированно упадет.
