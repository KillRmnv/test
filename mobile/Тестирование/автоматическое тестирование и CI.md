Интеграция автотестов в CI — это процесс переноса запуска тестов с твоего локального компьютера на удаленные сервера сборки (серверы автоматизации).

Вместо того чтобы нажимать кнопку "Run" в IntelliJ IDEA, запуск тестов доверяют CI-системе (GitLab CI, GitHub Actions, Jenkins), которая триггерится на изменения в репозитории.

## Как это устроено технически?

Интеграцию автотестов в CI можно разбить на четыре последовательных этапа.

### Шаг 1. Настройка сборщика проекта (Maven/Gradle)

Чтобы CI-сервер мог запустить тесты через консоль (у него нет графического интерфейса), проект должен быть правильно сконфигурирован. В Java-мире за это отвечают плагины в `pom.xml` (Maven) или `build.gradle` (Gradle) — например, **Maven Surefire Plugin** (для unit-тестов) и **Maven Failsafe Plugin** (для интеграционных тестов).

Благодаря им тесты запускаются простой терминальной командой:

Bash

```
mvn clean test -DsuiteXmlFile=testng.xml
```

### Шаг 2. Написание конфигурационного файла CI (Pipeline-as-Code)

В корень твоего проекта с автотестами кладется конфигурационный файл (например, `.gitlab-ci.yml` для GitLab или `main.yml` в папке `.github/workflows` для GitHub). В нем описывается, какую операционную систему запустить на сервере, какую версию Java поставить и какую команду выполнить.

Пример простого шага для запуска тестов в `.gitlab-ci.yml`:

YAML

```
stages:
  - test

run_regression_tests:
  stage: test
  image: maven:3.8.5-openjdk-17  # Docker-образ с Java и Maven
  script:
    - mvn clean test -DsuiteXmlFile=testng.xml  # Запуск наших тестов
  artifacts:
    when: always
    paths:
      - target/surefire-reports/  # Сохраняем отчеты TestNG после прогона
      - target/allure-results/    # Сохраняем результаты для Allure
```

### Шаг 3. Запуск инфраструктуры (Docker / Grid)

Если тесты проверяют базу данных или UI (через браузер), CI-серверу нужно предоставить эту инфраструктуру:

- **Для API / DB тестов:** В пайплайне временно поднимаются «сервисы» (например, контейнер с PostgreSQL и контейнер с тестируемым бэкендом).
    
- **Для UI тестов (Selenium/Playwright):** Браузеры запускаются в **headless-режиме** (без отрисовки графического окна на экране) прямо внутри Docker-контейнера на сервере сборки, либо тесты шлют запросы на удаленный браузерный хаб (Selenoid, Selenium Grid).
    

### Шаг 4. Публикация результатов

После завершения прогона CI-сервер парсит JUnit XML-отчеты.

Если какой-то тест упал, пайплайн завершается со статусом **Failed (красный)**. Если все тесты прошли успешно, статус становится **Success (зеленый)**.

Настроенный плагин (например, Allure-плагин для Jenkins) автоматически генерирует HTML-страницу с красивыми графиками прямо в интерфейсе сборки.

## Сценарии запуска автотестов в CI

В зависимости от целей тестирования, автотесты встраивают на разные этапы разработки:

1. **При каждом коммите / Pull Request (Smoke/Pre-merge):**
    
    Запускается легкий и быстрый набор тестов (обычно Unit-тесты и короткий Smoke на API — до 5-10 минут). Если они падают, разработчик не может влить свой код в основную ветку.
    
2. **По расписанию (Nightly Builds / Полный регресс):**
    
    Запуск тяжелых, долгих сквозных (E2E) и UI-тестов, которые могут идти часами. Пайплайн настраивается на запуск по cron-расписанию (например, каждую ночь в 02:00). Утром команда QA получает готовый отчет о стабильности всей системы.
    
3. **При деплое на тестовый стенд (Post-deploy):**
    
    Как только новый билд успешно развернут на QA-сервере, CI автоматически триггерит тесты, проверяющие работоспособность критических путей на этом стенде.
    

### 💡 Как ответить на интервью:

> _«Интеграция автотестов в CI строится на принципах Pipeline-as-Code. Мы настраиваем сборщик (например, Maven с плагином Surefire для запуска TestNG/JUnit), пишем конфигурационный YAML-файл для CI-системы, в котором описываем шаги запуска, требования к окружению (Java, Docker) и правила сбора артефактов (Allure-отчетов)._
> 
> _Обычно мы разделяем запуск: быстрые Smoke-тесты гоняем на каждый Pull Request, чтобы заблокировать плохой код на раннем этапе, а полный регрессионный пакет запускаем ночью по расписанию, чтобы не нагружать агенты сборки в рабочее время»_.
