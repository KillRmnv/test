Под «инструкциями» в контексте TestNG и JUnit обычно понимают **аннотации** (annotations) — специальные маркеры в Java-коде, которые начинаются с символа `@`. Именно они указывают тестовому движку, как обрабатывать методы, в каком порядке их запускать и какие настройки применять.

Оба фреймворка имеют очень похожую структуру жизненного цикла теста, но синтаксис аннотаций немного различается.

Ниже приведена подробная карта соответствия аннотаций между **JUnit 5 (Jupiter)** и **TestNG**.

### 1. Аннотации управления жизненным циклом (Setup & TearDown)

Эти инструкции нужны для подготовки окружения (например, открыть соединение с БД, запустить браузер) и его последующей очистки.

|**Действие / Время запуска**|**JUnit 5**|**TestNG**|
|---|---|---|
|**Один раз перед всеми** тестами в текущем классе|`@BeforeAll` _(метод должен быть static)_|`@BeforeClass`|
|**Перед каждым** тестовым методом в классе|`@BeforeEach`|`@BeforeMethod`|
|**После каждого** тестового метода в классе|`@AfterEach`|`@AfterMethod`|
|**Один раз после всех** тестов в текущем классе|`@AfterAll` _(метод должен быть static)_|`@AfterClass`|

#### Эксклюзивные уровни конфигурации в TestNG:

TestNG имеет более глубокую иерархию управления, которой нет в JUnit:

- `@BeforeSuite` / `@AfterSuite`: Запускается один раз до/после выполнения всего набора тестов (suite), описанного в `testng.xml`. Идеально для глобального старта Docker-контейнеров или инициализации ReportPortal.
    
- `@BeforeTest` / `@AfterTest`: Запускается до/после тестов, объединенных в тег `<test>` в файле `testng.xml`.
    
- `@BeforeGroups` / `@AfterGroups`: Запускается до/после выполнения определенной группы тестов (например, только перед тестами из группы `"database"`).
    

### 2. Базовые инструкции выполнения тестов

|**Назначение**|**JUnit 5**|**TestNG**|**Описание**|
|---|---|---|---|
|**Сам тест**|`@Test`|`@Test`|Помечает метод как тестовый сценарий.|
|**Игнорирование**|`@Disabled`|`@Test(enabled = false)`|Временно отключает тест. Он не будет запускаться при прогоне.|
|**Таймаут**|`@Timeout(value = 5)`|`@Test(timeOut = 5000)`|Тест упадет по ошибке, если его выполнение займет больше указанного времени.|

### 3. Инструкции для работы с данными (Параметризация)

Параметризация позволяет запустить один и тот же тест с разными наборами данных, чтобы не плодить дублирующий код.

#### В JUnit 5:

Используется связка аннотации `@ParameterizedTest` со специальными источниками данных:

- `@ValueSource`: Передача простого массива (например, только строк или чисел: `@ValueSource(strings = {"admin", "user"})`).
    
- `@CsvSource` / `@CsvFileSource`: Передача табличных данных (например, пары логин + пароль).
    
- `@MethodSource`: Ссылка на Java-метод, который возвращает поток сложных объектов (Stream/Arguments).
    

Java

```
@ParameterizedTest
@ValueSource(ints = {1, 2, 3})
void testWithSimpleArray(int argument) {
    assertTrue(argument > 0);
}
```

#### В TestNG:

Используется более мощный и классический для автоматизации механизм **Data Provider**:

- `@DataProvider`: Помечает метод, который генерирует и возвращает массив объектов с тестовыми данными.
    
- `@Test(dataProvider = "name")`: Связывает тест с конкретным провайдером данных.
    

Java

```
@DataProvider(name = "loginData")
public Object[][] getData() {
    return new Object[][] {
        {"standard_user", "secret_sauce"},
        {"locked_out_user", "secret_sauce"}
    };
}

@Test(dataProvider = "loginData")
public void loginTest(String username, String password) {
    // Код теста, который выполнится 2 раза с разными парами данных
}
```

### 4. Специфические фичи TestNG (Почему его любят автоматизаторы)

В аннотации `@Test` в TestNG встроено множество параметров, которые в JUnit 5 реализуются сложнее:

- **Зависимости между тестами:**
    
    `@Test(dependsOnMethods = {"login"})` — тест не начнется, пока успешно не пройдет тест `login`. Если `login` упадет, этот тест автоматически получит статус _Skipped_ (пропущен).
    
- **Группировка:**
    
    `@Test(groups = {"smoke", "api"})` — позволяет помечать тесты тегами и гибко запускать их через `testng.xml`.
    
- **Параллелизация на уровне метода:**
    
    `@Test(threadPoolSize = 3, invocationCount = 9)` — TestNG запустит этот тест 9 раз в 3 параллельных потока. Очень удобно для стресс-тестирования или проверки на состояние гонки (race conditions).
    

### 💡 Как ответить на интервью:

> _«В обоих фреймворках базовый жизненный цикл настраивается схожим образом через Before/After аннотации, но есть терминологические различия: например, JUnit 5 использует `@BeforeEach`/`@AfterEach`, а TestNG — `@BeforeMethod`/`@AfterMethod`._
> 
> _Ключевое отличие заключается в гибкости настроек самого теста. В TestNG аннотация `@Test` сама по себе является мощным инструментом конфигурации — прямо в ней можно настроить зависимости между методами через `dependsOnMethods`, сгруппировать тест с помощью `groups` или задать поставщика данных через параметр `dataProvider`. В JUnit 5 для этого чаще требуются отдельные специализированные аннотации»_.
