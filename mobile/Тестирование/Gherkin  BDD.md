### 1. BDD: Это про общий язык, а не про тесты

**BDD (Behavior-Driven Development)** — это не инструмент и не фреймворк. Это **процессный подход** к разработке, который фокусируется на общении.

- **Суть:** BDD помогает команде (продакты, разработчики, тестировщики) говорить на одном языке. Вместо сухих технических требований, все обсуждают **поведение системы** на примерах из реальной жизни.
- **Результат:** Эти обсуждения превращаются в **живую документацию** и **автоматизированные тесты**, которые всегда актуальны.

**Ключевая мысль для собеседования:** Gherkin — это язык, на котором мы записываем результаты BDD-обсуждений.

### 2. Gherkin: Синтаксис и Ключевые Слова

Gherkin — это язык, который легко читается и понимается любым человеком, даже далеким от программирования. Он использует структуру **Given-When-Then**.

#### Структура `.feature` файла:

*   **`Feature:`** (Функция/Функционал) — описывает, какой большой блок функциональности мы тестируем.
*   **`Scenario:`** (Сценарий) — конкретный пример поведения системы в рамках этой функции.
*   **`Given` (Дано/Допустим)**: Описывает **начальное состояние** системы перед тем, как пользователь что-то сделает.
*   **`When` (Когда/Если)**: Описывает **действие** пользователя или внешней системы.
*   **`Then` (Тогда/То)**: Описывает **ожидаемый результат** этого действия.
*   **`And` (И/Также)**: Позволяет добавить дополнительные условия к `Given`, `When` или `Then`.
*   **`But` (Но/А)**: Используется для отрицательных ожиданий (альтернатива `And`).

**Важно:** Каждый шаг (`Given`, `When`, `Then`) в `feature`-файле должен быть сопоставлен с **Java-кодом** (это называется Step Definition) с помощью аннотаций `@Given`, `@When`, `@Then`.

### 3. Живой пример: Банкомат (ATM)

Лучший способ понять Gherkin — посмотреть на пример. Возьмем классическую историю про снятие денег в банкомате.

**`src/test/resources/atm/withdrawal.feature`**:

```gherkin
Feature: Снятие наличных в банкомате
  Как пользователь банкомата
  Я хочу снимать наличные со своего счета
  Чтобы иметь доступ к деньгам в любое время

  Scenario: Успешное снятие денег
    Given на счету пользователя есть 100 долларов
    And карта пользователя действительна
    When пользователь запрашивает снятие 20 долларов
    Then банкомат должен выдать 20 долларов
    And баланс на счету должен стать 80 долларов
```
*Этот сценарий понятен и менеджеру, и разработчику, и тестировщику.*

### 4. Связь с Java (Ваша суперсила!)

Ваш опыт Java-разработчика здесь дает вам огромное преимущество. Каждая строчка из `feature`-файла должна быть реализована в Java-коде — это и есть **автоматизация**.

**`src/test/java/atm/StepDefinitions.java`** (упрощенный пример):
```java
package atm;

import io.cucumber.java.en.Given;
import io.cucumber.java.en.When;
import io.cucumber.java.en.Then;
import static org.junit.Assert.assertEquals;

public class StepDefinitions {

    private Account account = new Account();

    @Given("на счету пользователя есть {int} долларов")
    public void на_счету_пользователя_есть_долларов(int initialBalance) {
        account.setBalance(initialBalance);
    }

    @Given("карта пользователя действительна")
    public void карта_пользователя_действительна() {
        // Логика проверки валидности карты
    }

    @When("пользователь запрашивает снятие {int} долларов")
    public void пользователь_запрашивает_снятие_долларов(int amount) {
        account.withdraw(amount);
    }

    @Then("банкомат должен выдать {int} долларов")
    public void банкомат_должен_выдать_долларов(int expectedAmount) {
        // Проверка, что банкомат выдал expectedAmount
    }

    @Then("баланс на счету должен стать {int} долларов")
    public void баланс_на_счету_должен_стать_долларов(int expectedBalance) {
        assertEquals(expectedBalance, account.getBalance());
    }
}
```

### 5. "Бонусный" уровень: `Scenario Outline` (Сценарий c Примерами)

Когда у вас есть один и тот же сценарий, но с разными данными, не пишите кучу одинаковых `Scenario`. Используйте **`Scenario Outline`**.

```gherkin
  Scenario Outline: Снятие денег с разными суммами
    Given на счету пользователя есть <начальный_баланс> долларов
    And карта пользователя действительна
    When пользователь запрашивает снятие <сумма_снятия> долларов
    Then банкомат должен выдать <сумма_снятия> долларов
    And баланс на счету должен стать <конечный_баланс> долларов

    Examples:
      | начальный_баланс | сумма_снятия | конечный_баланс |
      | 100              | 20           | 80              |
      | 50               | 50           | 0               |
      | 1000             | 300          | 700             |
```
Этот один сценарий автоматически превратится в три теста с разными наборами данных!

### 6. Лучшие практики (Чтобы блеснуть на собеседовании)

Упомяните эти моменты, чтобы показать, что вы не просто знаете синтаксис, но и понимаете, как использовать Gherkin правильно:

1.  **Пишите на простом, нетехническом языке:** Избегайте технического жаргона в `feature`-файлах. Их цель — быть понятными для бизнеса.
2.  **Один сценарий — одно поведение:** Сценарии должны быть короткими и сфокусированными на одной конкретной бизнес-логике.
3.  **Переиспользуйте шаги (Steps):** Java-методы с аннотациями `@Given`, `@When`, `@Then` можно использовать в разных сценариях. Это упрощает поддержку кода.
4.  **Разделяйте логику:** Step Definition должен только связывать Gherkin-шаг с вызовом нужных методов вашего приложения. Вся сложная логика должна быть вынесена в отдельные классы (Page Object, Service Layer).

**Резюмируем для ответа на собеседовании:**

*   BDD — это про **коммуникацию и общий язык** в команде.
*   Gherkin — это инструмент (язык) для записи этого общего языка, использующий структуру **Given-When-Then**.
*   Gherkin-сценарии хранятся в **`.feature`** файлах.
*   Каждый шаг в Gherkin связывается с **Java-кодом** через Cucumber.
*   Для проверки разных данных используется **`Scenario Outline`** с блоком **`Examples`**.
*   Это делает тесты **живой документацией**, которую понимает вся команда.

**Совет:** Если у вас есть возможность, скачайте IntelliJ IDEA, установите плагин **Cucumber for Java** и напишите пару своих первых `feature`-файлов. Практический опыт — лучшее доказательство на собеседовании.
