Продвинутых техник в реляционных СУБД десятки. Ниже — структурированный обзор, сгруппированный по задачам. В скобках указана типичная поддержка в современных версиях PostgreSQL, MySQL, SQL Server, Oracle.

---

### 📊 1. Продвинутые индексы и оптимизация
| Техника | Суть | Когда применять | Поддержка |
|--------|------|----------------|-----------|
| **Partial / Conditional Index** | Индекс только для строк, удовлетворяющих `WHERE` | Частые запросы к подмножеству данных (`WHERE status='active'`) | PG, SQL Server, Oracle |
| **Covering Index (`INCLUDE`)** | Индекс хранит все нужные столбцы, исключает lookup в таблицу | Выборки `SELECT a,b WHERE c=...`, частые `INDEX ONLY SCAN` | PG, SQL Server, Oracle |
| **Functional / Expression Index** | Индекс по результату функции или выражения | `LOWER(email)`, `JSONB_PATH`, вычисляемые поля | PG, Oracle, MySQL 8.0+ |
| **Специализированные индексы** | BRIN (упорядоченные большие данные), GIN/GiST (JSON, полнотекст, геометрия) | Телеметрия, логи, поиск по JSONB/тексту/координатам | PG, Oracle (Spatial/Text), MySQL (FULLTEXT) |
| **Расширенная статистика** | MCV, n_distinct, correlation hints, auto-tuning | Планировщик ошибается в cardinality, сканы заменяются на seq | PG (`CREATE STATISTICS`), Oracle (`DBMS_STATS`), SQL Server (Auto Create/Update Stats) |

---

### 🔍 2. Расширенный SQL и аналитика
| Техника | Суть | Пример сценария |
|--------|------|-----------------|
| **Оконные функции** | Вычисления по "окну" строк без `GROUP BY` | `ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY created_at DESC) = 1` (последняя сессия) |
| **Рекурсивные CTE** | Обход иерархий и графов | Дерево категорий, оргструктура, маршруты |
| `LATERAL` JOIN | Коррелированный подзапрос в `FROM`, эффективный `TOP-N per group` | "Последние 3 заказа каждого клиента" за один проход |
| **Materialized Views** | Физическое кэширование результата запроса | Агрегаты для дашбордов, обновление по расписанию или инкрементально |
| **Pattern Matching (`MATCH_RECOGNIZE`)** | Поиск последовательностей строк (SQL:2016) | Обнаружение мошеннических паттернов, анализ сессий | Oracle, PG (через расширение `pg_match_recognize`), Databricks SQL |

---

### 🏗️ 3. Архитектура и моделирование данных
| Техника | Суть | Нюансы |
|--------|------|--------|
| **Temporal Tables (SQL:2011)** | Автоматическое хранение истории изменений с системными периодами | Встроенно в SQL Server, Oracle, DB2. В PG реализуется триггерами или `pg_temporal` |
| **JSON/NoSQL внутри РБД** | Хранение полуструктурированных данных + реляционные возможности | `JSONB` (PG), `JSON` (MySQL). Индексация по ключам, валидация схем (JSON Schema) |
| **Columnar Storage / HTAP** | Колончатое хранение для аналитики поверх транзакционной БД | SQL Server Columnstore, Oracle In-Memory, PG + `citus`/`columnar` |
| **Logical Sharding** | Горизонтальное разделение таблицы по ключу на разные ноды | Требует маршрутизации (Citus, Vitess, ProxySQL). Усложняет JOIN и транзакции |
| **Table Inheritance / Polymorphic Design** | Наследование таблиц или EAV-подобные структуры | Наследование есть в PG, но часто заменяется партиционированием + `CHECK` |

---

### 🔒 4. Конкурентность и транзакции
| Техника | Суть | Когда полезно |
|--------|------|---------------|
| **Snapshot Isolation / MVCC** | Чтение не блокирует запись, каждая транзакция видит срез данных | По умолчанию в PG, Oracle, SQL Server (RCSI). Снижает contention |
| **Optimistic Locking** | Проверка версии/таймстампа при `UPDATE`, а не блокировка строк | Высокая конкуренция на обновление, редкие конфликты |
| **Advisory Locks** | Прикладные блокировки, не привязанные к строкам/таблицам | Синхронизация фоновых задач, rate-limiting, координаторы |
| **Savepoints** | Точка отката внутри транзакции | Retry-логика, частичный откат при ошибках в пакетной обработке |
| **Distributed 2PC / XA** | Атомарные транзакции across multiple DBs/shards | Только когда строго необходима согласованность. Дорого, сложно в recovery |

---

### 🔄 5. Репликация, CDC и потоки данных
| Техника | Суть | Применение |
|--------|------|------------|
| **Logical Replication / CDC** | Потоковая передача изменений (INSERT/UPDATE/DELETE) в реальном времени | Синхронизация с DWH, Elasticsearch, Kafka, микросервисы |
| **Read Replicas + Connection Routing** | Автоматическое перенаправление `SELECT` на реплики | Масштабирование чтения, HA |
| **Logical Decoding / Change Streams** | Низкоуровневый доступ к WAL/binlog на уровне приложения | Кастомные пайплайны, аудит, event sourcing |
| **Conflict Resolution** | Стратегии разрешения конфликтов при multi-master репликации | `LAST_WRITE_WINS`, custom merge, CRDTs (BDR, Tungsten, MySQL Group Replication) |

---

### 🛡️ 6. Безопасность и комплаенс
| Техника | Суть | Поддержка |
|--------|------|-----------|
| **Row-Level Security (RLS)** | Фильтрация строк на уровне роли/пользователя | PG, Oracle (VPD), SQL Server (RLS) |
| **Column Encryption / TDE** | Шифрование "на лету" или прозрачное шифрование хранилища | SQL Server Always Encrypted, Oracle TDE, PG `pgcrypto`/`libpq` SSL |
| **Dynamic Data Masking** | Автоматическое скрытие чувствительных данных в результатах | SQL Server, Oracle, MySQL 8.0+ (частично) |
| **Audit Triggers / System Audit** | Автоматический лог DML/DDL, compliance-отчёты | Встроенные audit-фреймворки, `pgAudit`, SQL Server Audit |

---

### 📈 7. Наблюдаемость и обслуживание
| Техника | Суть | Инструменты |
|--------|------|-------------|
| **Query Store / `pg_stat_statements`** | История планов, регрессии, топ медленных запросов | SQL Server Query Store, PG `pg_stat_statements`, MySQL Performance Schema |
| **Auto-Vacuum / Auto-Analyze Tuning** | Настройка порогов под паттерн нагрузки | PG `autovacuum_*`, MySQL `innodb_stats_persistent` |
| **Connection Pooling & Multiplexing** | Переиспользование соединений, транзакционный режим | PgBouncer, ProxySQL, RDS Proxy, Pgpool-II |
| **Tiered Storage / S3 Offload** | Перенос холодных партиций в объектное хранилище | PG 15+ `pg_repack` + S3, Oracle ZFS/ASM, MySQL `innodb_file_per_table` + archiving |

### 💡 Как выбирать и с чего начать?
1. **Определите боль**: медленные запросы → индексы/оконные функции/CBO tuning; растущая таблица → партиционирование/архивация; частые `UPDATE`-конфликты → optimistic locking/MVCC tuning.
2. **Не усложняйте ради сложности**: многие "продвинутые" фичи добавляют операционные накладные расходы (sharding, 2PC, custom CDC).
3. **Тестируйте под реальной нагрузкой**: планы меняются, статистика устаревает, пулинг соединений может менять семантику транзакций.
4. **Используйте встроенное**: современные облачные РБД (Aurora, Cloud SQL, Azure SQL, Oracle Autonomous) автоматизируют часть этих техник (auto-indexing, query tuning, storage tiering).
