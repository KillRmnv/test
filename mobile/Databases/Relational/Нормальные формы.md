Нормализация реляционных данных — это последовательный процесс устранения избыточности и аномалий обновления. Каждая следующая форма **строже предыдущей** и гарантирует определённый уровень целостности.

Ниже приведены все основные нормальные формы (1NF–6NF) с практическими примерами, указанием, когда форма нарушается намеренно, и как это соотносится с JPA/ORM.

---

## 🔹 1NF (Первая нормальная форма)
**Правило:** Все атрибуты атомарны. В одной ячейке — одно значение. Нет повторяющихся групп. Есть первичный ключ.
```sql
-- ❌ Нарушение
employees(id, name, phones)
1, 'Иван', '111, 222, 333'

-- ✅ 1NF
employees(id, name)
employee_phones(emp_id, phone)
```
📌 **Практика:** Обязательное требование любой реляционной СУБД. JPA `@ElementCollection` или отдельные `@OneToMany` таблицы реализуют 1NF автоматически.

---

## 🔹 2NF (Вторая нормальная форма)
**Правило:** `1NF` + все неключевые атрибуты **полностью** зависят от всего составного первичного ключа (нет частичных зависимостей).
```sql
-- ❌ Нарушение (ключ: order_id, product_id)
order_items(order_id, product_id, product_name, quantity)
-- product_name зависит только от product_id

-- ✅ 2NF
order_items(order_id, product_id, quantity)
products(product_id, product_name)
```
📌 **Практика:** Актуальна только для таблиц с составными ключами. В ORM обычно решается через `@IdClass` / `@EmbeddedId` + вынос сущностей.

---

## 🔹 3NF (Третья нормальная форма)
**Правило:** `2NF` + нет **транзитивных зависимостей**. Каждый неключевой атрибут зависит только от ключа, целиком от ключа и ни от чего кроме ключа.
```sql
-- ❌ Нарушение
employees(emp_id, name, dept, dept_manager)
-- dept_manager зависит от dept, а dept зависит от emp_id (транзитивная зависимость)

-- ✅ 3NF
employees(emp_id, name, dept_id)
departments(dept_id, dept_name, manager_id)
```
📌 **Практика:** **Стандарт для OLTP-систем**. Большинство Spring/JPA проектов стремятся к 3NF. `SINGLE_TABLE` в JPA намеренно нарушает 3NF ради производительности.

---

## 🔹 BCNF (Нормальная форма Бойса–Кодда)
**Правило:** Усиление 3NF. Любая детерминанта (левая часть функциональной зависимости) обязана быть **суперключом**.
```sql
-- ❌ Нарушение BCNF (но проходит 3NF)
enrollments(student, course, instructor)
FD: (student, course) → instructor
FD: instructor → course  -- instructor не ключ, но определяет course

-- ✅ BCNF
student_instructor(student, instructor)
instructor_course(instructor, course)
```
📌 **Практика:** Теоретически строже 3NF, но на практике нарушается редко. JPA редко требует явной декомпозиции до BCNF, если нет сложных бизнес-ограничений.

---

## 🔹 4NF (Четвёртая нормальная форма)
**Правило:** `BCNF` + нет **нетривиальных многозначных зависимостей** (MVD), когда два атрибута независимы друг от друга при одном ключе.
```sql
-- ❌ Нарушение (комбинаторный взрыв)
emp_skills_lang(emp_id, skill, language)
-- Сотрудник знает {Java, Python} и {RU, EN} → 4 строки вместо 2+2

-- ✅ 4NF
emp_skills(emp_id, skill)
emp_langs(emp_id, language)
```
📌 **Практика:** Решает проблему "перекрёстного дублирования". В JPA маппится как два независимых `@ElementCollection` или `@OneToMany`.

---

## 🔹 5NF (Пятая нормальная форма / PJNF)
**Правило:** `4NF` + таблица не может быть无损 (losslessly) разбита на более мелкие, которые затем соединяются обратно без потери информации. Учитывает **зависимости соединения** (Join Dependencies).
```sql
-- Пример: supplier_part_project(supplier, part, project)
-- Если бизнес-правило гласит: "Поставщик поставляет деталь для проекта, 
-- только если он уже поставляет эту деталь И деталь используется в проекте",
-- возникает зависимость соединения.
```
📌 **Практика:** В реальных проектах почти не применяется явно. Достигается автоматически при корректном моделировании связей `@ManyToMany` и отсутствии скрытых бизнес-ограничений, связывающих 3+ сущности.

---

## 🔹 6NF (Шестая нормальная форма)
**Правило:** Таблица не допускает никаких нетривиальных зависимостей соединения. Каждая таблица содержит только **ключ + один атрибут** (или интервал времени).
```sql
-- Пример для темпоральных данных:
emp_salary(emp_id, valid_from, valid_to, amount)
emp_dept(emp_id, valid_from, valid_to, dept_id)
```
📌 **Практика:** Используется в **temporal DB**, DWH, аудит-системах. Для OLTP почти всегда денормализуют обратно из-за стоимости `JOIN`.

---

## 📊 Сводная таблица

| Форма | Что устраняет | Практическое применение | Типичное нарушение в JPA |
|-------|--------------|-------------------------|--------------------------|
| **1NF** | Неатомарные значения, повторяющиеся группы | Обязательно для любой реляционной БД | `@ElementCollection` без отдельной таблицы |
| **2NF** | Частичные зависимости от составного ключа | Таблицы связей, справочники | `@IdClass` с лишними полями |
| **3NF** | Транзитивные зависимости | Стандарт OLTP, большинство приложений | `SINGLE_TABLE`, встраивание справочных полей |
| **BCNF** | Детерминанты, не являющиеся ключами | Сложные бизнес-ограничения | Редко, обычно решается ограничениями БД |
| **4NF** | Многозначные зависимости (независимые списки) | Профили, теги, языки, навыки | Объединение `@ElementCollection` в одну таблицу |
| **5NF** | Зависимости соединения (3+ сущности) | Теоретическая строгость, DWH | Сложные `@ManyToMany` с доп. атрибутами |
| **6NF** | Любые зависимости соединения | Темпоральные данные, аудит, CQRS read-модели | Версионирование сущностей без разделения по атрибутам |

---

## 🔗 Связь с JPA и практикой разработки
1. **JPA по умолчанию стремится к 3NF** через отдельные таблицы для сущностей и связей.
2. **`@Inheritance(SINGLE_TABLE)`** намеренно нарушает 3NF (много `NULL`, денормализация) ради скорости полиморфных запросов.
3. **`@Embedded` / `@ElementCollection`** могут нарушать 4NF, если в одну таблицу помещаются независимые коллекции.
4. **Денормализация в продакшене** — норма. Для read-heavy систем создают материализованные представления, денормализованные DTO или используют CQRS. Нормализация остаётся эталоном для write-модели.

---

## ✅ Когда останавливаться?
| Сценарий | Рекомендуемая форма |
|----------|---------------------|
| OLTP, транзакционные системы, Spring Data JPA | **3NF** (иногда BCNF) |
| Отчётность, аналитика, DWH | **1NF–2NF** + контролируемая денормализация (звезда/снежинка) |
| Аудит, темпоральные данные, event sourcing | **6NF** (или темпоральные таблицы БД) |
| Прототипы, high-load read API | Часто **0NF–1NF** + кэш, денормализованные JSONB-поля |

> 💡 **Золотое правило:** Нормализуйте до тех пор, пока аномалии обновления не исчезнут. Затем денормализуйте **только там, где измеренные метрики (QPS, latency) требуют этого**, и документируйте причины отклонения.
