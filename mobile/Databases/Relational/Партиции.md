# 🗃️ Партиционирование в SQL: что это и зачем?

## 🔍 Что такое партиционирование?
**Партиционирование (partitioning)** — это механизм разбиения одной логически большой таблицы на несколько физических или логических частей (**партиций**), которые СУБД обрабатывает как единый объект. Для приложения и SQL-запросов таблица остаётся одной, но внутри данные хранятся и обрабатываются раздельно.

---

## 🎯 Зачем нужно?

| Цель | Как помогает |
|------|--------------|
| **⚡ Ускорение запросов** | СУБД пропускает нерелевантные партиции (`partition pruning`). Запрос к 10 ГБ данных может сканировать только 1 ГБ. |
| **🧹 Удобное управление данными** | Удаление/архивация старых данных через `DROP PARTITION` (мгновенно, без `DELETE` и генерации undo/WAL). |
| **💾 Распределение I/O** | Партиции можно размещать на разных дисках/табличных пространствах, снижая contention. |
| **🔧 Проще обслуживание** | `REINDEX`, `ANALYZE`, `VACUUM`, бэкапы можно делать по частям, не блокируя всю таблицу. |
| **📈 Параллелизм** | Современные СУБД могут параллельно сканировать несколько партиций. |

---

## 🧩 Основные типы партиционирования

| Тип | Критерий разбиения | Пример использования |
|-----|-------------------|----------------------|
| **Range** | Диапазон значений | По дате (`2023-01..2023-03`), по ID (`1..100000`) |
| **List** | Конкретные значения | По региону (`'EU'`, `'US'`), по статусу (`'active'`, `'archived'`) |
| **Hash** | Хеш-функция от ключа | Равномерное распределение, когда нет естественного диапазона |
| **Composite** | Комбинация типов | `RANGE(год) → LIST(регион)` для многоуровневой организации |

## 📝 Как это работает на практике (пример)
```sql
-- PostgreSQL (декларативное партиционирование)
CREATE TABLE orders (
    id serial,
    order_date date,
    amount numeric
) PARTITION BY RANGE (order_date);

CREATE TABLE orders_2023 PARTITION OF orders
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');
CREATE TABLE orders_2024 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
```
При выполнении:
```sql
SELECT * FROM orders WHERE order_date >= '2024-06-01';
```
Планировщик автоматически отсечёт `orders_2023` и будет сканировать только `orders_2024` (и будущие партиции, если есть).

---

## ✅ Когда применять
- Таблица **> 10–50 млн строк** или **> 10–20 ГБ**
- Есть чёткий **критерий разделения** (дата, регион, тип клиента и т.д.)
- Часто удаляются/архивируются **старые данные**
- Запросы обычно фильтруются по **ключу партиционирования**

## ❌ Когда не стоит
- Маленькие таблицы (< 1–2 млн строк)
- Запросы часто `JOIN`-ятся без учёта ключа партиционирования
- Частые cross-partition сканирования или отсутствие фильтров по ключу
- Слишком мелкое дробление (сотни/тысячи партиций) → рост накладных расходов планировщика
- Жёсткие требования к `UNIQUE`/`PRIMARY KEY` (в большинстве СУБД ключ партиционирования обязан входить в ограничение)

---

## ⚠️ Важные нюансы
1. **Не заменяет индексы**. Часто создают **локальные индексы** на каждой партиции.
2. **Partition pruning** работает только если в `WHERE` есть условие по ключу партиционирования.
3. **Реализация отличается** по СУБД:
   - PostgreSQL: декларативное с v10, развитое, но FK/unique имеют ограничения
   - MySQL: v5.7/8.0 поддерживает RANGE/LIST/HASH/KEY, но менее гибкое
   - SQL Server / Oracle: зрелые реализации, включая автоматическое управление партициями
4. **Over-partitioning** вредит: каждый план запроса учитывает все партиции, растёт время парсинга и потребления памяти.

---

## 📌 Краткий итог
Партиционирование — это **инфраструктурный инструмент для больших таблиц**. Оно даёт выигрыш в скорости выборки, упрощает жизненный цикл данных и обслуживание, но требует:
- правильного выбора ключа,
- дисциплины в написании запросов,
- мониторинга количества и размера партиций.
