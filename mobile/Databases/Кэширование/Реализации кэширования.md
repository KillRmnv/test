Ниже приведено практическое сравнение основных провайдеров кэша 2-го уровня, которые можно подключить к JPA/Hibernate через JCache (JSR-107). Сравнение ориентировано на **реальную эксплуатацию**, а не на маркетинговые характеристики.

### 📊 Сводная таблица

| Критерий | **Ehcache 3** | **Hazelcast** | **Infinispan** | **Redis (через JCache)** |
|:---|:---|:---|:---|:---|
| **Архитектура** | Embedded (в JVM) | Embedded или Client-Server | Embedded или Client-Server (Data Grid) | Внешний сервер (клиент-сервер) |
| **Кластеризация** | ❌ Нет в OSS (только коммерческий Terracotta) | ✅ Встроенная, auto-discovery, partitioning | ✅ JGroups, distributed/replicated/invalidation | ✅ Redis Cluster / Sentinel |
| **Хранение** | Heap → Off-heap → Disk | Heap (основное), есть off-heap & persistence | Heap + Off-heap + Persistent Store | RAM (+ RDB/AOF для persistence) |
| **Консистентность** | Strong (single-node), distributed weak | Strong distributed, настраиваемая репликация | ACID-подобная, JTA/XA, strong consistency | Eventual по умолчанию, настраивается |
| **Интеграция с Hibernate** | ⭐ Де-факто стандарт, нативный JCache | ✅ Хорошая (JCache), требуется адаптер | ✅ Отличная (исторически тесная связь с Hibernate) | ⚠️ Требует обёртки (`redisson-hibernate`, `hibernate-redis` или JCache-мост) |
| **Сложность эксплуатации** | Низкая | Средняя | Высокая | Средняя (если Redis уже в инфраструктуре) |
| **Лицензия** | Apache 2.0 | Apache 2.0 (Hazelcast 5+) | Apache 2.0 | SSPLv1/RSALv2 (Redis 7.2+), адаптеры — OSS |
| **Лучший сценарий** | Монолиты, один узел, быстрый старт | Микросервисы, общий кэш без внешней БД | Enterprise, строгая консистентность, Quarkus/WildFly | Polyglot-стек, уже есть Redis, high-load |

### 🔍 Детальный разбор провайдеров

#### 1. **Ehcache 3**
- ✅ **Плюсы:** Простейшая настройка, поддержка Tiered Storage (данные автоматически перетекают из heap → offheap → disk), отличная интеграция со Spring Boot и Hibernate 6, низкий оверхед.
- ❌ **Минусы:** В open-source **нет кластеризации**. Каждый узел держит свой независимый кэш. При обновлении сущности на одной ноде остальные не узнают об этом до истечения TTL или принудительной инвалидации.
- 🎯 **Когда брать:** Single-node приложения, справочники с TTL, проекты, где кэш должен работать «из коробки» без администрирования.

#### 2. **Hazelcast**
- ✅ **Плюсы:** Полноценный in-memory data grid. Узлы сами находят друг друга (multicast/TCP/Kubernetes), данные автоматически шардируются. Поддержка репликации, WAN-репликации, SQL поверх кэша.
- ❌ **Минусы:** Потребляет heap JVM, требует аккуратной настройки GC. В embedded-режиме увеличивает footprint приложения. Сложнее отлаживать расхождения данных при сетевых партициях.
- 🎯 **Когда брать:** Кластер микросервисов на JVM, нужен общий кэш без развёртывания отдельной инфраструктуры, высокая частота чтений по ID.

#### 3. **Infinispan**
- ✅ **Плюсы:** Исторически разрабатывался как L2-кэш для Hibernate. Поддерживает `invalidation`, `replicated` и `distributed` режимы. Полная поддержка JTA/XA, strong consistency, продвинутая обработка конфликтов.
- ❌ **Минусы:** Высокий порог входа, сложная конфигурация XML/программная, требует выделенных ресурсов для кластера. Документация местами фрагментирована.
- 🎯 **Когда брать:** Enterprise-системы, высокие требования к транзакционной консистентности, стек на базе Quarkus/WildFly, строгие SLA.

#### 4. **Redis (через JCache / Redisson)**
- ✅ **Плюсы:** Язык-независимый, выделенный процесс, готовые инструменты мониторинга (RedisInsight), горизонтальное масштабирование, persistence из коробки.
- ❌ **Минусы:** Hibernate не имеет официального модуля для Redis. Приходится использовать community-адаптеры или JCache-обёртку. Сетевые задержки снижают выигрыш для `READ_WRITE` стратегии. Изменение лицензии Redis (SSPL/RSAL) важно для коммерческого использования.
- 🎯 **Когда брать:** Redis уже есть в проекте, polyglot-архитектура, кэш должен переживать рестарты JVM, нагрузка измеряется десятками тысяч RPS.

### 🧭 Как выбирать: практический чек-лист

| Ваш контекст | Рекомендованный провайдер |
|:---|:---|
| Один сервер / монолит / нужен быстрый старт | `Ehcache 3` |
| Кластер из 3+ JVM, общий кэш без внешней БД | `Hazelcast` |
| Строгая консистентность, JTA, enterprise | `Infinispan` |
| Уже есть Redis, high-load, несколько языков | `Redis + Redisson/JCache` |
| Только локальный кэш на уровне сервиса (не JPA L2) | `Caffeine` (не используйте как L2 без обёрток) |

### ⚠️ Критичные нюансы при работе с Hibernate L2

1. **Стратегия конкурентного доступа + распределённый кэш**  
   `READ_WRITE` и `TRANSACTIONAL` требуют блокировок/координации. В распределённых кэшах это даёт заметный сетевой оверхед. Часто выгоднее использовать `NONSTRICT_READ_WRITE` + короткий TTL, если допустима микрорассинхронизация.
2. **Query Cache vs Entity Cache**  
   L2-кэш хранит только сущности по ID. Результаты `JPQL`/`Criteria` кэшируются отдельно через Query Cache. Он хранит только списки ID, а сами объекты всё равно берёт из L2 или БД. Включайте его осознанно: он быстро становится «грязным».
3. **Оптимистическая блокировка обязательна**  
   При `READ_WRITE`/`NONSTRICT_READ_WRITE` обязательно поле `@Version`. Без него Hibernate не сможет детектировать конфликты при параллельных обновлениях.
4. **Прямые SQL-модификации не инвалидируют кэш**  
   Если вы пишете `entityManager.createNativeQuery("UPDATE ...")` или используете миграции/админку в обход JPA, кэш останется «грязным». Нужно вручную вызывать `entityManager.getEntityManagerFactory().getCache().evict()` или использовать триггеры/шины событий.
5. **Мониторинг обязателен**  
   Включите `hibernate.generate_statistics=true` и следите за `second_level_cache_hit_count / request_count`. Если hit ratio < 60–70%, кэш скорее вредит (накладные расходы + память), чем помогает.

### 📦 Пример подключения (универсальный через JCache)
Независимо от провайдера, Hibernate 6 подключается одинаково:
```yaml
spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_second_level_cache: true
          region:
            factory_class: jcache # org.hibernate.cache.jcache.JCacheRegionFactory
      javax.cache.provider: <ваш JCache Provider> # напр. org.ehcache.jsr107.EhcacheCachingProvider
```
Далее кладёте `ehcache.xml`, `hazelcast-client.xml` или настраиваете Redis-адаптер в classpath. Логика работы с `@Cacheable` и `@org.hibernate.annotations.Cache` не меняется.

### ✅ Итог
- **Ehcache** → простота, один узел, быстрый старт  
- **Hazelcast** → кластер JVM, общий кэш без внешней инфраструктуры  
- **Infinispan** → enterprise, транзакции, строгая консистентность  
- **Redis** → polyglot, уже в стеке, высокая масштабируемость  

Если скажете ваш стек (кол-во узлов, частота записи/чтения, наличие Redis/другого кэша, требования к консистентности), подберу конкретный провайдер и дам готовый `application.yml` + настройки сущностей.
