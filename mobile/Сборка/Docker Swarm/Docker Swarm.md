**Docker Swarm** — это режим и инструмент для **объединения нескольких серверов (физических или виртуальных) в единый кластер**, превращая их в один мощный "виртуальный" компьютер для запуска контейнеров . Он нужен, чтобы сделать приложения **отказоустойчивыми** и **масштабируемыми** .

Если Docker Compose управляет контейнерами на одной машине, то Swarm управляет **сервисами** (группами одинаковых контейнеров), распределяя их по целому кластеру машин.

### Зачем нужен Docker Swarm? Основные задачи

Представь, что твое приложение стало популярным, и один сервер уже не справляется. Или этот сервер может неожиданно выйти из строя. Swarm решает эти проблемы:

1.  **Высокая доступность (High Availability) и отказоустойчивость**: Если один сервер (нода) в кластере выйдет из строя, Swarm автоматически перезапустит контейнеры с твоим приложением на других живых серверах. Пользователи ничего не заметят .
2.  **Масштабирование (Scaling)**: Ты можешь одним движением увеличить количество копий (реплик) своего сервиса. Допустим, было 3 контейнера с веб-сервером, а стало 10. Swarm сам распределит, на каких серверах кластера их запустить .
3.  **Балансировка нагрузки (Load Balancing)**: Входящие запросы к твоему приложению автоматически распределяются между всеми запущенными контейнерами по всему кластеру. При этом не важно, на какой сервер кластера пришел запрос — Swarm сам доставит его до нужного контейнера .
4.  **Обновления без простоя (Rolling updates)**: Ты можешь обновить версию приложения постепенно. Swarm будет по очереди останавливать старые контейнеры и запускать новые. Если что-то пойдет не так, он автоматически откатит изменения .

### Ключевые понятия

Чтобы понимать, как это работает, нужно знать несколько терминов :

*   **Узел (Node)**: Это сервер, который входит в состав кластера Swarm. Бывают двух типов:
    *   **Менеджер (Manager)**: "Мозг" кластера. Управляет состоянием, распределяет задачи, хранит конфигурацию. Для отказоустойчивости менеджеров должно быть нечетное количество (например, 3 или 5) .
    *   **Рабочий (Worker)**: "Рабочая лошадка". На них непосредственно запускаются контейнеры. Они получают задачи от менеджера и просто их выполняют .
*   **Сервис (Service)**: Это описание твоего приложения в кластере. Ты говоришь: "Хочу запустить 5 копий контейра `nginx:latest` с открытым портом 80". Swarm воспринимает это как "желаемое состояние" и постоянно следит за его выполнением .
*   **Задача (Task)**: Это единица работы, которую менеджер назначает рабочему узлу. Обычно одна задача = один запущенный контейнер .
*   **Стек (Stack)**: Группа сервисов, которые вместе образуют приложение. Описывается в уже знакомом тебе формате YAML (файл `compose.yml`), но с некоторыми дополнениями .

### Как это работает простыми словами?

Это похоже на создание команды. Сначала ты делаешь один сервер главным (инициализируешь swarm). Он становится **менеджером**. Потом даешь команду другим серверам: "присоединяйтесь к нам", и они становятся **рабочими** .

```bash
# На главном сервере (manager)
docker swarm init --advertise-addr <IP-менеджера>

# Эта команда выдаст токен для подключения рабочих узлов.
# На каждом рабочем сервере выполняешь:
docker swarm join --token <токен> <IP-менеджера>:2377
```

После этого у тебя есть кластер. Вместо `docker-compose up` ты используешь `docker stack deploy` .

```bash
# Развернуть стек сервисов, описанных в файле compose.yml
docker stack deploy -c compose.yml my_app
```

Docker Swarm сам решит, на каких узлах запустить контейнеры, и будет следить, чтобы они всегда работали.

### Docker Swarm vs. Kubernetes

Это главный вопрос выбора инструмента для оркестрации. Ты уже пользовался Docker, поэтому Swarm будет для тебя очень простым и понятным. Но на рынке есть и другой тяжеловес — Kubernetes .

**Сравнение в таблице**

| Характеристика | Docker Swarm | Kubernetes |
| :--- | :--- | :--- |
| **Сложность** | **Очень простой**. Встроен в Docker, использует те же команды CLI. Порог входа минимален . | **Сложный**. Требует изучения новых концепций и инструментов. Крутая кривая обучения . |
| **Установка и настройка** | Быстрая и легкая, буквально пара команд . | Долгая и трудоемкая, требует глубокого понимания . |
| **Масштабирование** | Быстрое, но ограниченное по сравнению с Kubernetes. | **Масштабируется практически бесконечно**, автоматически, подходит для самых больших нагрузок . |
| **Функциональность** | Базовый, но достаточный для большинства задач набор функций. | **Огромная экосистема**, автодополнение, самовосстановление, продвинутые стратегии обновлений . |
| **Для кого** | Идеален для небольших и средних проектов, стартапов, команд, которые уже используют Docker и хотят быстро получить отказоустойчивость . | Стандарт для enterprise-сегмента, крупных корпораций и проектов с очень высокой нагрузкой и сложной архитектурой . |

### Резюме

*   **Docker Swarm** — это встроенная в Docker "волшебная палочка", которая превращает набор серверов в единый супер-сервер для твоих контейнеров.
*   Он нужен, чтобы приложение не падало при отказе одного сервера и чтобы его можно было легко масштабировать, увеличивая количество копий.
*   Для тебя, после изучения Compose, переход на Swarm будет практически бесшовным — тот же YAML-файл, только команды немного меняются (`docker stack deploy` вместо `docker compose up`).
*   Kubernetes — это мощнее и сложнее. Swarm — это выбор для 99% проектов, где нужна надежность, но не хочется усложнять себе жизнь.
