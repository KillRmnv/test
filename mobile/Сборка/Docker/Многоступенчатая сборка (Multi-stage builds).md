Это мощная техника оптимизации `Dockerfile`, которая позволяет создавать очень маленькие и безопасные финальные образы. Идея проста: вы используете один `Dockerfile`, в котором описываете несколько этапов (стадий) сборки [](https://dou.ua/lenta/articles/building-docker/)[](https://hacksobesov.com/questions/golang-developer/chto-takoe-multi-stage-sborka-v-docker-i-zachem-ona-nuzhna/#content).

- **Этап 1 (Builder)**: Вы берете "толстый" образ со всеми инструментами для сборки (компилятор Go, SDK, Maven, и т.д.), копируете исходный код и собираете приложение. На этом этапе создаются все необходимые артефакты (например, скомпилированный бинарный файл).
    
- **Этап 2 (Final)**: Вы берете минимальный образ (например, `alpine`) и **копируете в него только результат сборки** (бинарный файл) с первого этапа. Все инструменты сборки, исходный код и временные файлы остаются в промежуточном образе и не попадают в финал.
    

### Как это выглядит в коде (пример для Go)

dockerfile

# ---- Этап 1: Сборка приложения ----
# Используем полный образ Golang (весом более 500 МБ) для компиляции
FROM golang:1.21-alpine AS builder
WORKDIR /app
# Копируем файлы зависимостей для лучшего кэширования
COPY go.mod go.sum ./
RUN go mod download
# Копируем исходный код и компилируем
COPY . .
RUN go build -o myapp .
# ---- Этап 2: Финальный образ ----
# Используем пустой Alpine (весом ~5 МБ) для запуска
FROM alpine:latest
WORKDIR /root/
# Магия: копируем ТОЛЬКО бинарник с первого этапа
COPY --from=builder /app/myapp .
# Запускаем приложение
CMD ["./myapp"]

**Что здесь происходит?**

1. **`AS builder`** дает имя первому этапу.
    
2. Инструкция **`COPY --from=builder`** копирует файл из образа, созданного на этапе `builder`, в текущий финальный образ.
    

### Зачем это нужно?

- **Радикальное уменьшение размера**: В примере выше финальный образ будет весить всего **~10-15 МБ**, хотя образ для сборки занимал более 500 МБ [](https://hacksobesov.com/questions/golang-developer/chto-takoe-multi-stage-sborka-v-docker-i-zachem-ona-nuzhna/#content). Инструменты для компиляции просто не попадают в результат.
    
- **Безопасность**: В финальном образе нет ни компилятора, ни исходного кода, ни лишних утилит. Это сильно сокращает потенциальную поверхность для атаки [](https://github.com/AlzoabiAF/Swarm)[](https://hacksobesov.com/questions/golang-developer/chto-takoe-multi-stage-sborka-v-docker-i-zachem-ona-nuzhna/#content).
    
- **Разделение окружений**: Вы можете использовать разные базовые образы для сборки и для запуска. Собрать можно на `maven:3.8-openjdk-17`, а запустить на `eclipse-temurin:17-jre-alpine` [](https://ru.werf.io/docs/v1.2.339/usage/build/stapel/imports.html).
    
- **Простота**: Раньше для такого трюка приходилось писать сложные bash-скрипты и использовать несколько `Dockerfile`. Теперь все делается в одном файле [](https://dou.ua/lenta/articles/building-docker/).
