## Зачем минимизировать количество слоев?

### 1. Скорость сборки и пуша

Каждый слой — это отдельная сущность, которую нужно:

- Создать при сборке
    
- Запаковать
    
- Отправить в registry (при push)
    
- Скачать (при pull)
    

Чем меньше слоев, тем быстрее все эти операции. Особенно это критично в CI/CD пайплайнах.

### 2. Размер образа

Каждый слой хранит изменения относительно предыдущего. Но если вы в одном слое что-то создаете, а в следующем удаляете — файлы всё равно остаются в предыдущем слое и занимают место!

### 3. Производительность при запуске

При запуске контейнера Docker объединяет все слои в единую файловую систему (используя UnionFS). Меньше слоев = быстрее монтирование.

## Антипаттерны и как их исправить

###  Плохо: Много отдельных RUN

dockerfile

RUN apt-get update
RUN apt-get install -y python3
RUN apt-get install -y python3-pip
RUN apt-get install -y curl
RUN apt-get clean
RUN rm -rf /var/lib/apt/lists/*

**Проблема:** 6 слоев вместо одного. И каждый слой хранит временные файлы apt.

###  Хорошо: Объединяем логически связанные команды

dockerfile

RUN apt-get update && \
    apt-get install -y python3 python3-pip curl && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

**Почему лучше:**

- Один слой вместо шести
    
- Временные файлы apt удаляются **в том же слое**, поэтому они не попадают в финальный образ
    

### ❌ Плохо: Копирование и удаление в разных слоях

dockerfile

COPY temp-file.txt .
RUN rm temp-file.txt

**Проблема:** Файл `temp-file.txt` был добавлен в первом слое, а удален во втором. Но в первом слое он всё еще существует! Docker просто "скрывает" его во втором слое, но место он занимает.

### ✅ Хорошо: Всё в одном слое

dockerfile

RUN echo "Some content" > temp-file.txt && \
    use_temp-file.txt && \
    rm temp-file.txt

## Продвинутые техники минимизации

### 1. Объединение RUN с умом

**Правило:** Объединяйте команды, которые связаны по смыслу и работают с одними и теми же временными файлами.

dockerfile

# Установка зависимостей Python
RUN pip install --no-cache-dir -r requirements.txt && \
    pip cache purge
# Установка пакетов и очистка кэша
RUN apt-get update && \
    apt-get install -y --no-install-recommends package1 package2 && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

### 2. Стратегический порядок COPY

Слои кэшируются. Если вы часто меняете код, но редко — зависимости, расположите COPY так:

dockerfile

# Сначала копируем только файлы зависимостей
COPY package.json package-lock.json ./
RUN npm install
# Потом копируем весь остальной код
COPY . .

**Почему:** Если вы измените код, но не `package.json`, Docker переиспользует слой с `npm install` из кэша.

### 3. Многоступенчатая сборка (о которой мы говорили)

Это не просто минимизация слоев, это полное разделение на этапы:

dockerfile

# Этап сборки (может быть много слоев)
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
# Финальный этап (минимум слоев)
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80

В финальном образе — всего 2-3 слоя, хотя в процессе сборки их могли быть десятки.
