Привет! Давай разберемся с работой над `Dockerfile`. Это основной инструмент для создания собственных образов контейнеров. Я подготовил подробное руководство, которое поможет тебе понять структуру, ключевые инструкции и процесс сборки.

###  Что такое Dockerfile?

`Dockerfile` — это просто текстовый документ, который содержит пошаговые инструкции для сборки образа контейнера . Его можно сравнить с рецептом: ты указываешь, какой взять базовый "ингредиент" (базовый образ), какие команды выполнить, какие файлы добавить и как запустить приложение. Docker автоматически выполняет эти инструкции, создавая готовый образ .

###  Ключевые инструкции Dockerfile

Вот основные команды, которые чаще всего используются при написании `Dockerfile` .

| Инструкция | Описание | Пример использования |
| :--- | :--- | :--- |
| **`FROM`** | Задает базовый образ, на котором будет основан ваш образ. Это **обязательная первая инструкция** . | `FROM python:3.13-slim` <br>`FROM mcr.microsoft.com/dotnet/runtime:6.0`  |
| **`WORKDIR`** | Устанавливает рабочую директорию для всех последующих инструкций (`COPY`, `RUN`, `CMD`). Если папка не существует, она будет создана . | `WORKDIR /app` <br>`WORKDIR /usr/local/app`  |
| **`COPY`** | Копирует файлы и папки из контекста сборки (твоего проекта) в файловую систему образа . | `COPY requirements.txt ./` <br>`COPY src ./src`  |
| **`ADD`** | Более продвинутая версия `COPY`. Помимо копирования, умеет распаковывать локальные архивы и скачивать файлы по URL . | `ADD app.tar.gz /app` (распакует архив) |
| **`RUN`** | Выполняет команды **во время сборки** образа. Обычно используется для установки пакетов, зависимостей и настройки окружения . | `RUN pip install --no-cache-dir -r requirements.txt` <br>`RUN apt-get update && apt-get install -y curl` |
| **`EXPOSE`** | Указывает, какой порт приложение будет слушать внутри контейнера. Это скорее документация, для публикации порта наружу используется флаг `-p` в `docker run` . | `EXPOSE 8080` <br>`EXPOSE 80`  |
| **`ENV`** | Устанавливает переменные окружения, которые будут доступны в контейнере . | `ENV NODE_ENV=production` |
| **`USER`** | Определяет пользователя, от имени которого будут выполняться все команды (`RUN`, `CMD`, `ENTRYPOINT`). Это повышает безопасность . | `USER app`  |
| **`CMD`** | Задает команду, которая будет выполняться **при запуске контейнера**. Может быть переопределена аргументами в `docker run`. В файле может быть только одна `CMD` . | `CMD ["node", "./src/index.js"]` <br>`CMD ["dotnet", "MyApp.dll"]`  |
| **`ENTRYPOINT`** | Настраивает исполняемый файл, который будет запускаться при старте контейнера. В отличие от `CMD`, её сложнее переопределить. Часто используется вместе с `CMD` для передачи аргументов по умолчанию . | `ENTRYPOINT ["dotnet"]`<br>`CMD ["myapp.dll"]` |

###  Процесс сборки образа

1.  **Создай файл:** В корне твоего проекта создай файл с именем `Dockerfile` (без расширения) .
2.  **Напиши инструкции:** Наполни файл инструкциями, следуя логике: `FROM` -> `WORKDIR` -> `COPY` -> `RUN` -> ... -> `CMD`.
3.  **Выполни сборку:** Открой терминал в папке с `Dockerfile` и выполни команду:
    ```bash
    docker build -t my-image-name:tag .
    ```
    *   `-t` задает имя и тег для образа.
    *   `.` (точка в конце) — это путь к контексту сборки (текущая папка), где лежит твой `Dockerfile` и файлы для копирования .
###  Полезные советы

*   **Оптимизация кэширования:** Располагай инструкции, которые меняются редко (установка зависимостей), в начале `Dockerfile`. Инструкции, которые меняются часто (копирование кода), — в конце. Это ускорит повторные сборки, так как Docker будет использовать кэш для неизменяемых слоев .
*   **Минимизируй размер:** Используй минималистичные базовые образы (например, `-alpine` или `-slim`) и очищай временные файлы в одной инструкции `RUN`, чтобы не раздувать образ .
*   **Безопасность:** Старайся не запускать контейнеры от имени `root`. Создавай отдельного пользователя с помощью `RUN useradd` и инструкции `USER` .
