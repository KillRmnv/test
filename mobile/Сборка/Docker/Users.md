## Опция `user` в Docker Compose
### Без `user` (по умолчанию)
Если не указать `user`, процесс запускается от **root** (UID 0) внутри контейнера. Это стандартное поведение Docker.

### С `user`
```yaml
services:
  myapp:
    image: myapp
    user: "1000:1000"  # UID:GID
    # или
    user: "john"       # имя пользователя (если существует в контейнере)
    # или
    user: "1000"       # только UID, группа будет по умолчанию
```

## Зачем это нужно?

### 1. **Безопасность** (главная причина)
Запускать приложения от root — плохая практика. Если злоумышленник взломает приложение через уязвимость, он получит root-доступ к контейнеру. Хотя это не даст ему root на хосте (из-за изоляции), но внутри контейнера он сможет делать всё что угодно.

**Принцип наименьших привилегий**: приложение должно иметь ровно столько прав, сколько нужно для работы, и не больше.

### 2. **Проблемы с правами доступа к файлам** (как в твоем примере)
Это самая частая практическая проблема. Когда ты монтируешь папку с хоста в контейнер (volumes):

```yaml
volumes:
  - ./surreal_data:/mydata
```

Файлы, созданные в контейнере, будут принадлежать пользователю, от которого запущен процесс в контейнере.

**Проблема:**
- На хосте файлы создаются от UID пользователя внутри контейнера
- Если в контейнере процесс от root (UID 0), то на хосте файлы будут принадлежать root
- Обычный пользователь на хосте не сможет их читать/писать/удалять без sudo

**Решение:**
Запускать процесс в контейнере от того же UID, что и пользователь на хосте.

### 3. **Совместимость с bind mounts**
В твоем примере комментарий гласит:
```yaml
user: root # Required for bind mounts on Linux
```

Это специфичный случай для SurrealDB. Некоторые приложения требуют root внутри контейнера, потому что:
- Им нужно открывать "привилегированные" порты (<1024)
- Им нужен доступ к системным вызовам
- Или (как в этом случае) для работы с bind mounts на Linux

## Форматы записи

```yaml
# Только UID (группа будет по умолчанию)
user: "1000"

# UID и GID
user: "1000:1000"

# Только имя пользователя (должен существовать в /etc/passwd контейнера)
user: "www-data"

# Имя пользователя и группа
user: "www-data:www-data"

# Числовой UID с именованной группой
user: "1000:www-data"
```

## Как узнать свой UID на хосте?
```bash
$ id -u
1000  # твой UID

$ id -g
1000  # твоя основная группа

$ id
uid=1000(roman) gid=1000(roman) groups=1000(roman),4(adm),27(sudo)...
```

## Практический пример

### Проблема: файлы создаются от root
```yaml
services:
  app:
    image: node:18
    user: root  # или не указан user
    volumes:
      - ./data:/app/data
    command: node create-file.js
```

После запуска:
```bash
$ ls -la data/
-rw-r--r-- 1 root root 123 файл.txt  # принадлежит root!
$ rm файл.txt
rm: remove write-protected file? # просит sudo
```

### Решение: использовать свой UID
```yaml
services:
  app:
    image: node:18
    user: "1000:1000"  # твой UID:GID
    volumes:
      - ./data:/app/data
    command: node create-file.js
```

Теперь:
```bash
$ ls -la data/
-rw-r--r-- 1 roman roman 123 файл.txt  # принадлежит тебе!
$ rm файл.txt  # работает без sudo
```

## Тонкости и нюансы

### 1. **Пользователь должен существовать в контейнере**
Если укажешь `user: "john"`, но в образе нет пользователя `john`, будет ошибка. Для числовых UID это не требуется — Linux разрешает процессы от любого UID, даже если его нет в /etc/passwd.

### 2. **Разные базовые образы = разные пользователи**
- **Alpine**: по умолчанию есть `root`, `nobody`, `daemon`
- **Ubuntu/Debian**: есть `www-data`, `nobody` и др.
- Многие образы создают своего пользователя (например, `node` для Node.js образов)

### 3. **Комбинация с USER в Dockerfile**
В Dockerfile тоже есть инструкция `USER`. Порядок приоритета:
1. `user` в compose.yaml **переопределяет** `USER` из Dockerfile
2. Можно указать в командной строке: `docker run --user 1000 ...`

### 4. **Группы и права доступа**
Иногда нужно добавить пользователя в дополнительные группы:
```yaml
user: "1000:1000"
```
Здесь первый `1000` — UID, второй `1000` — GID (основная группа).

## Почему в твоем примере user: root?

Вернемся к твоему коду:
```yaml
surrealdb:
  image: surrealdb/surrealdb:v2
  command: start --log info --user root --pass root rocksdb:/mydata/mydatabase.db
  user: root  # Required for bind mounts on Linux
  volumes:
    - ./surreal_data:/mydata
```

Здесь есть важный момент: SurrealDB с движком **rocksdb** требует определенных прав для работы с файловой системой на уровне ядра. Некоторые операции с bind mounts на Linux требуют root-привилегий внутри контейнера. Разработчики образа, зная это, указали, что для bind mounts нужен root.

Это компромисс между безопасностью и работоспособностью.

## Безопасная альтернатива

Если хочешь сохранить данные и не использовать root:

```yaml
services:
  surrealdb:
    image: surrealdb/surrealdb:v2
    user: "${UID:-1000}:${GID:-1000}"
    volumes:
      - ./surreal_data:/mydata
    # Остальные настройки...
```

И в `.env` файле:
```
UID=1000
GID=1000
```

Но в случае с SurrealDB/rocksdb это может не сработать из-за особенностей работы с файловой системой.
