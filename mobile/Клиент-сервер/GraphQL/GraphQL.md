## 🔷 Что такое GraphQL (строго)

**GraphQL — это:**

1. язык запросов к API
    
2. типизированная схема данных
    
3. среда выполнения этих запросов на сервере
Это **не протокол** и **не замена HTTP**. Обычно работает поверх HTTP.

---

## 🔷 Архитектура GraphQL

В GraphQL нет «контроллеров» и «эндпоинтов». Есть:

```
Schema → Types → Queries/Mutations → Resolvers → Data Sources
```

---

## 🔷 1. Schema (самое главное)

Schema — это **жёсткое типизированное описание API**.

Пример:

```graphql
type User {
  id: ID!
  name: String!
  email: String!
  orders: [Order]
}

type Order {
  id: ID!
  price: Float!
}

type Query {
  user(id: ID!): User
}
```

Это **контракт API**, как OpenAPI/WSDL, но для GraphQL.

---

## 🔷 2. Query — получение данных

Аналог GET, но гораздо гибче.

Запрос:

```graphql
query {
  user(id: 10) {
    name
    orders {
      price
    }
  }
}
```

Сервер вернёт **ровно** эти поля.

---

## 🔷 3. Mutation — изменение данных

Аналог POST/PUT/DELETE.

```graphql
mutation {
  createUser(name: "John", email: "john@example.com") {
    id
    name
  }
}
```

---

## 🔷 4. Resolver — магия GraphQL

Resolver — это функция на сервере, которая знает, **откуда взять данные**.

Например:

- user → из БД
    
- orders → из другого микросервиса
    
- email → из кеша
    

Клиент об этом не знает.

---

## 🔷 Почему GraphQL называют «агрегатором данных»

Один запрос может собрать данные:

- из БД
    
- из другого сервиса
    
- из кеша
    
- из внешнего API
    

И склеить в один ответ.

В REST это были бы 3–5 запросов.

---

## 🔷 Проблема REST, которую решает GraphQL

### ❌ Overfetching (слишком много данных)

REST:

```
GET /users/10
```

Вернул 20 полей, нужно 2.

### ❌ Underfetching (не хватает данных)

Нужно:

- user
    
- его заказы
    
- статусы заказов
    

В REST → несколько запросов.

GraphQL → один.

---

## 🔷 Как реально выглядит HTTP-запрос

```http
POST /graphql
Content-Type: application/json
```

Тело:

```json
{
  "query": "query { user(id: 10) { name email } }"
}
```

---

## 🔷 Типизация — огромный плюс

GraphQL строго типизирован:

- String
    
- Int
    
- Float
    
- Boolean
    
- ID
    
- свои типы
    

Ошибки ловятся на уровне схемы.

---

## 🔷 Интроспекция (уникальная фича)

Клиент может спросить у сервера:

> «Какие типы и запросы ты поддерживаешь?»

И на основе этого строятся:

- GraphiQL
    
- автодокументация
    
- автогенерация клиентов
    

---

## 🔷 Подписки (Subscriptions)

GraphQL поддерживает **реальное время** через WebSocket.

```graphql
subscription {
  newOrder {
    id
    price
  }
}
```

Это то, чего в REST вообще нет.

---

## 🔷 Когда GraphQL особенно хорош

- Микросервисы
    
- Мобильные приложения (экономия трафика)
    
- Сложные связи данных
    
- Когда фронт часто меняется
    
- Когда нужно агрегировать данные из разных мест
    

---

## 🔷 Когда REST лучше

- Простые CRUD API
    
- Кэширование через HTTP
    
- Публичные API
    
- Когда важна простота

> GraphQL — это язык запросов к API с типизированной схемой, позволяющий клиенту запрашивать только необходимые данные через единый эндпоинт, агрегируя информацию из различных источников и устраняя проблемы overfetching и underfetching, характерные для REST.
