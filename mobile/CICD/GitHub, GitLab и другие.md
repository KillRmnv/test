Сам по себе **Git** — это просто консольная программа, которая работает локально на твоем компьютере. Она отлично отслеживает историю изменений в файлах, но она изолирована. Если твой жесткий диск сгорит, история пропадет. Если друг захочет написать код вместе с тобой, ему будет сложно получить твои изменения.

**GitHub, GitLab, Bitbucket** и другие подобные сервисы — это «социальные сети» и облачные хранилища для разработчиков. Они берут мощь локального Git и добавляют к ней инструменты для командной работы и автоматизации.

Вот 4 главные задачи, которые они решают на реальных проектах:

### 1. Удаленное хранилище (Бэкап)

Платформа выступает единым источником правды (Single Source of Truth). Вся команда пушит (отправляет) свой локальный код на удаленный сервер. Если твой ноутбук сломается, ты просто скачаешь актуальную версию репозитория на новый и продолжишь работу с того же места.

### 2. Совместная разработка и Code Review

В enterprise-разработке никто не заливает код напрямую в главную ветку (`main` или `master`).

- Разработчик делает свою задачу в отдельной ветке, пушит ее на GitHub/GitLab и создает **Pull Request** (или Merge Request в терминологии GitLab).
    
- Это специальный интерфейс, где другие члены команды (и QA-инженеры) могут посмотреть разницу в коде (diff), оставить комментарии к конкретным строкам и одобрить (Approve) или отклонить изменения.
    

### 3. Автоматизация тестирования и доставки (CI/CD)

Это то, с чем ты будешь плотно работать в автоматизации.

Современные платформы имеют встроенные инструменты (GitHub Actions, GitLab CI), которые автоматически запускают скрипты при каждом пуше в репозиторий.

- _Пример:_ Ты написал тесты на Java в TestNG. Как только разработчик заливает новый код в GitLab, сервер _сам_ собирает проект, _сам_ запускает твои тесты и, если хотя бы один упал, блокирует слияние этого кода с главной веткой.
    

### 4. Управление проектом (Issue Tracking)

Тебе не обязательно нужен отдельный Jira, чтобы вести проект. На этих платформах есть встроенные трекеры задач: там можно заводить баг-репорты (Issues), назначать исполнителей, вешать теги (баг, фича) и привязывать исправление конкретного бага к конкретному коммиту.

### Краткое сравнение популярных платформ

|**Платформа**|**Главный фокус**|**Инструмент CI/CD**|**Популярность в индустрии**|
|---|---|---|---|
|**GitHub**|Open Source, социальное взаимодействие, самое большое IT-сообщество.|GitHub Actions|Стандарт для открытых проектов и стартапов.|
|**GitLab**|Enterprise-сегмент, концепция «All-in-one DevOps» из коробки.|GitLab CI/CD (очень мощный)|Часто ставится на собственные закрытые сервера компаний (on-premise).|
|**Bitbucket**|Тесная интеграция с продуктами Atlassian (Jira, Confluence).|Bitbucket Pipelines|Корпоративный сектор, сидящий на экосистеме Atlassian.|

### 💡 Как красиво ответить на интервью:

> _«Git — это локальная система контроля версий, инструмент для отслеживания изменений в файлах. А GitHub и GitLab — это веб-сервисы, которые предоставляют облачный хостинг для Git-репозиториев. Их главная цель — обеспечить удобную совместную разработку через механизм Pull/Merge реквестов, интегрировать процесс Code Review и автоматизировать сборку и тестирование кода с помощью встроенных CI/CD конвейеров»_.
